# ゲームプレイUI全面リファクタリング点検

作成日時：2026-07-18 JST

## 変更概要

- ゲーム開始後はゲーム画面だけを表示。
- 自分を下に固定した四方向の相対座席。
- 4枚の場札を卓の中央へコンパクトに重ね、座席・実況との接触を回避した一画面構成。
- ピック専用の集中表示。
- CPU実況を最新1件へ整理。
- ログとルールをサイドパネル化。
- PC、スマホ縦、スマホ横、高さの低いPC、幅360px以下へ対応。

## レスポンシブ設計確認

| 想定画面 | 主な調整 |
|---|---|
| 1366×768以上 | 大型カード、広い中央卓 |
| 1024×768 | 標準カード、四方向席 |
| 1024×600 | カードと手札領域を縮小 |
| 390×844 | スマホ縦向き、中央卓の高さを優先 |
| 320×568 | 小型座席・カード・中央円へ縮小 |
| 844×390 | スマホ横向き、手札高とカード高を圧縮 |
| 667×375 | 横向き最小クラスの高さへ対応 |

## 実施した検証

- `server.js` JavaScript構文確認。
- クライアントJavaScriptをAcornで解析。
- CSSをCSSTreeで解析。
- HTMLをparse5で解析し、構文エラーなし。
- 実DOMのID重複なし。
- 代表7画面サイズで中央場札の占有幅・高さを数値検証し、すべて卓内へ収まることを確認。
  - 1366×768 / 1024×768 / 1024×600
  - 390×844 / 320×568 / 844×390 / 667×375
- プレイ画面の動的HTMLを単体検証。
  - 自分が下へ配置される。
  - 4人分の座席が生成される。
  - 4方向のトリックスロットが生成される。
  - 中央に時計回り表示が生成される。
  - CPU吹き出しが座席と重複しない。
- ピック画面の動的HTMLを単体検証。
  - ピック専用モードへ切り替わる。
  - 候補枚数分の裏向きボタンが生成される。
  - 勝者と最弱が表示される。
- WebSocket実進行テスト。
  - 1人＋CPU3人で1ラウンドを開始。
  - トリック、5秒レビュー、候補選択、ピック、ペア処理を経てゲーム終了。
  - 最終得点を生成し、121秒で正常完走。
- 3枚パス＋開始時ペア捨てONの組み合わせで、両フェイズを経て通常プレイへ到達。

## 検証環境上の制約

- この作業環境のクラウドブラウザではローカルURLとデータURLの表示が制限され、スクリーンショットによるピクセル比較は実施できなかった。
- 代替として、HTML/CSS/JavaScriptの構文解析、動的描画単体テスト、代表画面寸法の収まり計算、WebSocket実進行テストを実施した。

## 維持した仕様

- サーバー権威型の`playableCardIds`。
- マストフォロー、勝者、最弱判定。
- ピック候補選択とランダム配置。
- CPUが裏向きカードの内容を見ない公平性。
- ババブタ、マッド・ピッグ、シュート・ザ・ピッグ。
- 得点計算、ラウンド、再接続、進行停止監視。
